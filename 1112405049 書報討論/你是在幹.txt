第一章：緒論 (Chapter 1: Introduction)今日的企業環境以廣泛使用各式各樣的網頁應用程式為特徵，包括 ERP(前端)、CRM(後端)、獨立會計軟體、人資管理系統及 POS(交易) 系統 。一個常見的挑戰是，這些應用程式——特別是那些在不同時代開發或由不同廠商提供的系統——缺乏互操作性（Interoperability）。在這種情況下，營運人員往往嚴重依賴試算表軟體來彌補系統間的鴻溝，手動將資料從匯出的試算表複製並貼上到各種網頁應用程式中 。這種重複且容易出錯的過程既耗時又耗費資源 。自動化從試算表檔案中提取資料並隨後將其輸入網頁應用程式，可以顯著提高營運效率 。然而，許多企業網頁應用程式是外部開發的，缺乏可供直接整合的現成 API 。這創造了一個技術障礙，在傳統上，如果沒有大量的客製化開發努力或昂貴的整合平台，很難克服這個障礙 。

1.1 AI 代理的新興角色 (1.1 The Emerging Role of AI Agents)人工智慧的最新進展，特別是大型語言模型（LLM），為這個長期存在的挑戰提供了一種新穎的方法 。這些模型在理解與生成程式碼，以及解讀結構化規格書方面展現了卓越的能力 。隨著 AI 持續演進，我們正目睹能夠以極少人工干預執行複雜任務的 AI 代理（AI Agents）的出現 。AI 代理自主與網頁應用程式互動的概念，代表了企業自動化的範式轉移（Paradigm Shift）。與其依賴傳統的基於 UI 的自動化或 API 整合，這些代理有可能直接從機器可讀的規格書中理解網頁應用程式的底層結構與需求 。這種方法通過允許 AI 動態生成並執行必要的互動腳本，繞過了許多傳統的整合挑戰 。

1.2 研究缺口與機會 (1.2 Research Gap and Opportunity)儘管 LLM 與 AI 代理極具潛力，但在如何系統性地將這些技術應用於網頁應用程式自動化方面仍存在巨大缺口 。目前的做法通常涉及以下幾種：模擬使用者互動但脆弱且維護密集的瀏覽器自動化工具 。由於需要完全渲染使用者介面並執行 JavaScript 而效率低下的瀏覽器自動化工具 。需要大量開發資源的客製化開發整合解決方案 。在沒有結構化方法或框架的情況下，隨機使用 AI 生成腳本 。目前缺少的是一個能夠連結現有網頁應用程式與現代 AI 系統能力的全方位框架 。這樣一個框架需要能系統性地對網頁應用程式進行逆向工程、生成結構化規格，並使 AI 代理能夠解讀這些規格，以實現無需人工干預的工作流程自動化 。

1.3 研究目標 (1.3 Research Objectives)本論文旨在開發並驗證一個用於逆向工程網頁應用程式的框架，以產出可驅動 AI 代理自動化的軟體規格書 。具體而言，研究目標包括：設計一套在無需存取原始碼的情況下，系統性地從網頁應用程式中擷取互動模式與資料需求的技術 。開發一種結構化規格格式，以機器可讀的格式擷取網頁應用程式互動的核心要素 。展示現代 LLM 如何解讀這些規格以生成功能性的自動化腳本 。透過實際的網頁應用程式驗證該框架的有效性 。

1.4 重要性與未來意義 (1.4 Significance and Future Implications)這項研究的重要性延伸到了立即的自動化效益之外 。隨著 AI 代理變得更加精密，它們將越來越需要標準化的方式來理解並與現有的軟體系統互動 。這項研究願景了一個未來，即企業網頁應用程式將包含專用的規格區塊，其意圖不是供人類使用者使用，而是供 AI 代理使用 。這樣的未來將實現不同應用程式之間的無縫整合，而無需傳統的 API 開發，大幅減少目前跨系統工作流程所需的手動工作 。此外，它還將為遺留系統（Legacy Systems）提供一條過渡路徑，使其無需進行根本性的重新設計即可參與現代、AI 驅動的自動化生態系統 。


第二章：相關工作 (Chapter 2: Related Work)網頁應用程式的自動化，以及從黑箱系統（Black-box Systems）生成機器可讀規格書的過程，交織了多個研究領域，包括網頁逆向工程、機器人流程自動化（RPA）、企業應用整合（EAI），以及大型語言模型（LLMs）在軟體工程中的應用 。

2.1 網頁應用程式逆向工程 (2.1 Web Application Reverse Engineering)網頁應用程式的逆向工程自早期依賴靜態 HTML 分析以來，已有顯著的演進 。WebScarab 和 Burp Suite 等工具引入了流量檢查，作為在無需存取原始碼的情況下分析應用程式行為的手段 。Felmetsger 等人 [1] 的研究展示了流量模式如何揭示邏輯漏洞，為動態參數的結構化分析鋪平了道路 。最近，Mesbah 等人 [2] 提出了 DOM 狀態追蹤來分析基於 Ajax 的應用程式，突顯了用戶端互動的複雜性 。此後，HAR（HTTP Archive）檔案已成為擷取與分析 HTTP 層級行為的業界標準 [3] 。這些產出物允許檢查連續的「請求-回應」模式，這對於發現動態參數之間的因果關係至關重要 。Păsăreanu 等人 [4] 採用的方法則整合了從請求流中進行模型推論，以重建使用者工作流與依賴鏈 。

2.2 參數擷取與動態分析 (2.2 Parameter Extraction and Dynamic Analysis)理解參數如何在不同請求之間演變對於自動化至關重要 。傳統方法依賴於 DOM (文件物件模型)爬取和脆弱的 XPath(搜索法) 規則 [5] 。Minamide [6] 最近在 HTML 與 JavaScript 靜態分析方面的工作，以及動態汙點分析技術 [7]，都提高了準確性 。本論文引入的「基於配方（Recipe-based）」方法建立在這些想法之上，將擷取模式抽象化為通用模板，允許 AI 系統在不同應用程式之間解讀並重新生成此類邏輯 。

2.3 企業應用整合與互通性 (2.3 Enterprise Application Integration and Interoperability)企業應用整合仍是一個長期存在的挑戰 。Linthicum [8] 將整合策略進行了分類，從直接 API 到面向訊息的預設中間件 。然而，許多遺留應用程式缺乏 API，被迫通過 UI 自動化進行整合 。機器人流程自動化（RPA）透過 UiPath 和 Automation Anywhere [9][10] 等工具解決了這個問題，但這些工具通常很脆弱且需要大量的維護 。本論文的獨特性在於利用因果分析和依賴圖構建來建立穩定的自動化工作流 。

2.4 自動化規格書生成 (2.4 Specification Generation for Automation)機器可讀規格書的概念並不新鮮 。WSDL 和 OpenAPI 倡議試圖記錄 API [11][12]，而 BDD 框架中的 Gherkin 等 DSL（領域特定語言）則試圖記錄預期的系統行為 [13] 。然而，沒有一個直接解決從不透明系統中建立規格的問題 。本文提出的使用結構化配方生成基於 YAML 的規格書，在互動擷取與 AI 可取用邏輯之間引入了一座新穎的橋樑 。

2.5 大型語言模型於軟體工程 (2.5 Large Language Models in Software Engineering)Codex [14]、Code Llama [15] 和 Gemini [16] 等 LLM 的興起為自動化引入了新的可能性 。它們現在可以跨語言和框架生成可執行程式碼、解釋指令並完成任務 。ReAct [17] 和 Toolformer [18] 等框架證明，當提供足夠的結構時，LLM 可以通過工作流進行推理 。本論文建立在此基礎上，提供規格書作為這些模型的解讀計畫 。

2.6 目前方法的局限性 (2.6 Limitations in Current Approaches)儘管有所進展，現有工具在不同應用程式的通用化方面仍面臨挑戰 。TestRigor [19] 和 Selenium IDE [20] 實現了工作流自動化，但在具有動態生成 Token 或複雜重新導向鏈的工作階段中會失敗 。很少有系統嘗試自動擷取因果鏈，或將使用者輸入與應用程式產生的資料分開——這兩項挑戰正是本研究透過配方抽象化和使用者輸入啟發式方法直接解決的 。

2.7 綜述與研究貢獻 (2.7 Synthesis and Research Contribution)透過結合流量分析、模式分類、依賴建模和 AI 提示詞設計，本研究填補了為缺乏 API 的企業系統自動化工作流的空白 。所提出的框架與眾不同之處在於，它不僅擷取網頁應用程式的邏輯，還產出 AI 可讀的規格書，從而實現端到端的自動化 。


第三章：研究方法 (Chapter 3: Methodology)
本研究提出一個系統性框架，旨在將手動操作網頁應用程式的過程轉換為機器可讀的規格書，進而驅動 AI 代理（AI Agents）自動化。本章將詳細說明該框架的架構、演算法以及實作細節。

3.1 系統概述 (3.1 System Overview)
該框架由五個主要階段組成，旨在將原始的 HTTP 流量轉化為功能齊全的自動化腳本：

資料擷取與預處理：過濾並分析 HTTP 存檔（HAR）檔案。

參數分析與因果推導：識別動態參數及其來源。

依賴圖構建：建立各個請求之間的邏輯關係模型。

規格書生成：產出結構化的 YAML 文件。

AI 驅動的程式碼生成：將規格書轉譯為可執行的 Python 腳本。


3.2 資料擷取與預處理 (3.2 Data Capture and Preprocessing)
自動化流程始於擷取使用者與目標網頁應用程式互動時產生的 HTTP 流量。

3.2.1 HAR 檔案分析 (3.2.1 HAR File Analysis)
HTTP 存檔（HAR）格式用於記錄瀏覽器與伺服器之間交換的所有互動。每個 HAR 檔案包含一個「項目（Entries）」列表，代表一個請求與回應對。框架會解析這些項目以提取關鍵資訊，包括 URL、方法（GET/POST）、請求標頭、查詢參數、POST 資料以及回應主體（Response Body）。

3.2.2 流量過濾 (3.2.2 Traffic Filtering)
並非所有的 HTTP 請求對於自動化都是必要的。為了減少雜訊，框架應用了以下過濾規則：

資源類型過濾：排除圖片（PNG, JPG, GIF）、樣式表（CSS）、字體與靜態媒體檔案。

網域過濾：僅保留目標應用程式網域內的請求，排除第三方追蹤器、廣告或 CDN 資源。

重複移除：識別並移除不影響系統狀態的重複冗餘請求。

3.3 參數分析與因果推導 (3.3 Parameter Analysis and Causation Inference)
這是框架中最關鍵的組件，負責識別哪些資料是動態的，以及這些資料是如何在應用程式中傳遞的。

3.3.1 變量辨識 (3.3.1 Variable Identification)
框架會分析請求中的所有參數。參數被歸類為以下兩種類型之一：

靜態參數 (Static Parameters)：在多次操作中保持不變的值。

動態參數 (Dynamic Parameters)：隨工作階段（Session）或輸入而改變的值，例如 CSRF Token、Session ID 或系統產生的 ID。

3.3.2 因果關係分析 (3.3.2 Causation Analysis)
為了自動化工作流程，系統必須知道一個動態參數的值是從哪裡來的。框架會掃描先前的所有回應主體（Response Bodies），以尋找當前請求參數中出現的值。

3.4 規格書生成 (3.4 Specification Generation)
一旦確定了依賴關係，框架就會生成一份軟體規格書。本研究採用 YAML (YAML Ain't Markup Language) 格式，因為它在人類可讀性與機器解析性之間取得了平衡。規格書包含：

端點資訊 (Endpoint Information)：URL 與 HTTP 方法。

擷取規則 (Extraction Rules)：定義如何從回應中提取資料的配方（Recipes）。

有效負載定義 (Payload Definitions)：如何構建後續請求的結構。

3.5 AI 代理整合 (3.5 AI Agent Integration)
最後一階段涉及將生成的規格書輸入大型語言模型（LLM）。透過提示工程（Prompt Engineering），模型被指示將這些結構化指令轉換為使用特定程式庫（如 requests 或 playwright）的 Python 程式碼。這確保了生成的腳本具備正確的錯誤處理機制與狀態管理。

3.6 基於 AI 的腳本生成 (3.6 AI-Based Script Generation)框架的最後一個組件利用現代大型語言模型（LLMs），從前述階段建立的規格書中自動生成可執行的自動化腳本 。這項能力代表了網頁自動化領域的重大進展，實現了將結構化規格轉換為功能性程式碼且無需人工干預的過程 。為此目的，研究評估了多個頂尖的 AI 模型，包括 OpenAI 的 GPT-4o-mini 與 GPT-3o-mini、Anthropic 的 Claude 3.5 Haiku、Claude 3.5 Sonnet 與 Claude 3.7 Sonnet，以及 Google 的具備思考模式的 Gemini Flash 與 Gemini Pro 2.5 。雖然各模型的表現有所差異，但所有模型都展現了生成具備不同準確度與完整度的工作腳本的能力 。一項有趣的發現是，儘管框架包含了精心設計的提示詞（Prompts）來引導腳本生成，但即使是相對簡單的提示詞也往往能產出正確的腳本 。這顯示現代 LLM 在 HTTP 互動、網頁自動化以及與網頁爬取及表單提交相關的 Python 程式設計範式方面，已具備充實的知識 。值得注意的是，雖然腳本生成是本框架的一項寶貴產出，但更廣泛的意義在於證明了機器可讀的規格書能使 AI 代理自主地與網頁應用程式進行互動 。這指向了一個未來：網站可能會包含專門為 AI 代理設計的規格區塊，而非僅供人類使用者使用 。

3.6.1 LLM 整合架構 (3.6.1 LLM Integration Architecture)（此部分涉及模型串接的技術細節）3.6.2 精密的提示工程 (3.6.2 Sophisticated Prompt Engineering)提示詞經過刻意設計，旨在引導 AI 生成完整且具備功能的程式碼，而非解釋或部分實作 。這種對具體實作的關注有助於克服某些模型傾向提供教育性內容而非實際解決方案的趨勢 。3.6.3 未來擴展 (3.6.3 Future Extensions)基於 AI 的腳本生成組件在建構時已考慮到擴展性，允許未來進行多項增強以強化其穩健性與適應性 ：回饋迴路機制：整合回饋機制，使系統能測試生成的腳本並根據執行結果反覆改進品質 。多模型整合：透過多模型協作方法，利用不同 AI 系統的多元優勢來產出更可靠的自動化結果 。領域特定微調：預想透過微調模型行為以適應特定應用，提升準確度與情境理解力 。AI 代理專用格式：開發專為 AI 代理設計的規格格式是關鍵的一步，這些格式應優先考慮機器可讀性與結構化互動語義，而非以人類為中心的表現方式 。


第四章：結果與討論 (Chapter 4: Results and Discussion)本章透過將提出的逆向工程框架實際應用於真實世界的網頁系統來進行評估 。本研究採用基於案例研究的方法，專注於實際應用性 。評估內容包括框架是否能有效分析具備不同架構的複雜網頁、識別 HTTP 請求間的資料依賴性、生成反映實際應用行為的準確規格書，並產出能複製使用者互動的功能性自動化腳本 。4.1 案例研究實作 (4.1 Case Study Implementation)選擇了四個不同的網頁應用程式來評估框架在不同技術棧與複雜度下的能力 ：Moodle 學習管理系統 (moodle2.ntust.edu.tw)：學術門戶網站 。台灣鐵路訂票系統 (tip.railway.gov.tw)：運輸服務 。武陵農場露營訂位系統 (camping.wuling-farm.com.tw)：預約系統 。監理服務網 (mvdis.gov.tw)：政府服務 。4.3 綜合評估與意義 (4.3 Integrated Evaluation and Implications)所提出的框架在公共網頁應用程式與企業內部系統中皆展現了一致的表現，驗證了其在多元環境下的適應性 。對四個目標系統的評估顯示，參數提取成功率極高（80-100%），且規格書覆蓋完整 。特別是 Moodle 案例說明了框架簡化工作流複雜度的能力，在 24 個 HTTP 節點中僅識別出 3 個對完成任務至關重要的關鍵節點 。


第五章：結論與未來展望 (Chapter 5: Conclusion and Future Work)
5.1 結論 (5.1 Conclusion)
本研究成功開發並驗證了一個系統性框架，能夠對網頁應用程式進行逆向工程，並產出機器可讀的規格書以驅動 AI 代理（AI Agents）自動化。透過將焦點從脆弱的用戶端 UI 自動化轉向穩健的伺服器端 HTTP 互動分析，本框架克服了傳統自動化工具面臨的許多長期挑戰。

本研究的主要發現與貢獻總結如下：

自動化因果分析：研究證明了透過分析 HTTP 流量中的時間序列，可以在無需存取原始碼的情況下，精準識別網頁應用程式中的動態參數及其來源。

規格書驅動的範式：引入基於 YAML 的軟體規格書作為中介層，為現有系統與現代 AI 模型之間搭建了橋樑。這種方法能讓 AI 代理在具備明確結構指引的情況下進行推理。

LLM 的效能驗證：實驗結果顯示，當前的大型語言模型（如 GPT-4 與 Claude 3.5）具備足夠的程式碼理解能力，能將結構化規格書轉換為高品質、具備錯誤處理能力的 Python 腳本。

實務可行性：透過 Moodle、台鐵訂票系統等多個案例研究，證實了該框架能應對具備不同技術架構與複雜度的真實世界應用程式。

總體而言，本論文證明了逆向工程與 AI 技術的結合，能顯著降低企業流程自動化的技術門檻與維護成本，尤其是在缺乏 API 的環境下。

5.2 未來展望 (5.2 Future Work)
雖然本研究為 AI 驅動的自動化奠定了基礎，但仍有數個領域值得進一步探索：

複雜安全性處理：未來的研究可專注於自動化處理更高級的安全機制，例如多因素驗證（MFA）、生物識別或基於行為分析的抗機器人挑戰。

自我修復自動化：開發一套回饋機制，使 AI 代理在腳本執行失敗時，能自動重新分析流量並更新規格書，實現系統的自我修復。

多代理協同：探索由多個 AI 代理分工合作的架構（例如：一個負責分析、一個負責生成、一個負責測試），以處理更大型且跨系統的企業級工作流。

標準化與合規性：隨著 AI 代理的使用日益增加，研究如何制定專為 AI 設計的網頁規格標準，並在自動化過程中確保資料隱私與合規性將變得至關重要。
